## Decision: build #1 вҶ’ `hooksig`

Picked webhook signature verify/sign because:

1. Highest mention count (14/22 postings touched it explicitly or implicitly).
2. **No clean polyglot CLI exists.** Every existing OSS is either (a) a provider SDK that drags in the whole vendor lib, or (b) a framework-locked middleware (Express, Hono, FastAPI, DRF, Gin). A devops/platform person debugging a misbehaving webhook delivery in a CI job, a Go service, or a one-off shell pipeline has nothing.
3. Fits the вүҲ150 LOC budget (final: ~155 LOC source + ~90 LOC tests, stdlib only).
4. Strong BR signal вҖ” programathor #33452 (AE Digital, SГӘnior Node) explicitly lists "criar e gerenciar integraГ§Гөes com serviГ§os externos (pagamentos, e-mail, **webhooks**)" + "garantir idempotГӘncia e resiliГӘncia em operaГ§Гөes crГӯticas" as the day-to-day responsibilities. Per the rule "BR-bias when in doubt" this clinched it.
5. Replay-window handling (Stripe/Slack tolerance) is the part most hand-rolled implementations get wrong; baking it in protects future-me.

## Out of scope (deliberate)

- **BullMQ dashboard / DLQ replay** вҖ” already 5+ active OSS competitors (`bull-board`, `taskforce.sh`, `muleta`, `matador`, `bullbar`). Marked **solved**, skipped.
- **Feature-flag CLI** вҖ” `basestack-flags-cli` exists and is active; OpenFeature ecosystem already strong. Skipped.
- **Idempotency-key store** as a separate library вҖ” already covered per-framework (`gin-idempotency`, `hono-idempotency`, FastAPI, DRF). `hooksig` deliberately stays at the *signature verification* layer; idempotency belongs in the caller's storage layer.
- **Provider auto-detection from header shape** вҖ” would be cute, but explicit `provider` argument is safer (no ambiguity between Shopify base64 and generic base64).
